# 연결 끊김 정리 (연결되어야 하는데 연결 안 된 부분)

아래는 목록↔상세, 링크↔대상, 데이터 소스 불일치 등 **연결이 끊긴 부분**만 정리한 목록입니다.  
**(아래 항목들은 2026-02 기준 수정 완료됨)**

---

## 1. SUNO 워드 → 프로젝트 상세 (잘못된 URL)

**위치:** `src/routes/suno/words/+page.svelte`  
**현상:** 워드 카드에서 "연결된 트랙"을 클릭하면 `href="/suno/projects/{trackId}"` 로 이동한다.  
`trackId`는 `track1`, `track2`, … 이고, SUNO 프로젝트 상세는 `proj1`, `proj2`, `proj3` 같은 **프로젝트 id**를 쓴다.  
→ `/suno/projects/track1` 같은 URL은 상세에서 매칭되지 않고, 기대한 프로젝트로 연결되지 않음.

**필요 조치:**  
- 트랙 id → 프로젝트 id 매핑을 두고 `href="/suno/projects/{projectId}"` 로 연결하거나,  
- “연결된 트랙”이 트랙 상세를 의미한다면 `href="/tracks/{trackId}"` 로 변경하고, 트랙 상세/목록과 id 체계를 맞추기.

---

## 2. SUNO 대시보드 "최근 활동" → 프로젝트 상세 (링크 없음)

**위치:** `src/routes/suno/+page.svelte`  
**현상:** `recentActivities`에 `projectId`, `projectTitle`이 있지만, 각 활동 행에 **클릭 링크가 없음**.  
→ "최근 활동"에서 프로젝트로 이동할 수 없음.

**필요 조치:**  
각 활동 항목을 `<a href="/suno/projects/{activity.projectId}">` 로 감싸거나, 제목/프로젝트명에 해당 링크 적용.

---

## 3. 트랙 목록 ↔ 트랙 상세/편집 (데이터 소스 불일치)

**위치:**  
- 목록: `src/routes/tracks/+page.svelte` (인라인 목업: id 1=Demo Track, 2=Evening Breeze, …)  
- 상세: `src/routes/tracks/[id]/+page.svelte` (다른 인라인 목업: id 1=Sugar Rush, 2=Sweet Dreams, … 13=Summer Night)

**현상:**  
- 같은 id(예: 1)라도 목록과 상세의 제목/아티스트 등이 다름.  
- 목록에서 "편집"으로 `/tracks/1/edit` 갔다가 상세로 가면, 상세 쪽 id 1 데이터(Sugar Rush)가 보임.  
→ **목록에서 클릭한 트랙과 상세/편집에서 보이는 트랙이 일치하지 않음.**

**필요 조치:**  
트랙 목록·상세·편집이 **동일한 API 또는 동일한 목업 소스**를 쓰도록 통일하고, id별로 한 트랙만 가리키게 하기.

---

## 4. 태스크 목록 ↔ 태스크 상세 (id 미반영)

**위치:**  
- 목록: `src/routes/tasks/+page.svelte` (id 1~4, 제목 등 목업)  
- 상세: `src/routes/tasks/[id]/+page.svelte`

**현상:**  
상세 페이지가 **params.id를 쓰지 않고** 항상 같은 목업만 표시함 (예: "앨범 커버 디자인 검토").  
→ 목록에서 id 2, 3, 4를 눌러도 상세는 동일한 한 건만 보여줌.

**필요 조치:**  
상세에서 `params.id`로 해당 태스크를 조회해 표시 (API 연동 시 `GET /api/tasks/:id` 등).

---

## 5. (참고) 앨범 목록 ↔ 앨범 상세

**위치:**  
- 목록: `src/routes/albums/+page.svelte` (id 1,2,3 등)  
- 상세: `src/routes/albums/[id]/+page.svelte` (같은 id 체계로 더 많은 목업)

**현상:**  
같은 id(1,2,3)에 대해 목록과 상세가 같은 id를 쓰고 있어, **연결 자체는 동작**함.  
다만 목록/상세가 **각각 인라인 목업**을 갖고 있어, 추후 API로 바꿀 때 한쪽만 바꾸면 불일치 가능.

**권장:**  
앨범도 트랙처럼 **한 데이터 소스(API/목업)** 로 통일하는 것이 안전함.

---

## 6. (참고) 검색 API → 태스크 상세

**위치:** `src/routes/api/search/+server.ts`  
**현상:** 검색 결과에 `/tasks/11`, `/tasks/12` 등이 포함될 수 있으나, 태스크 목록/상세는 id 1~4만 다루고, 상세는 id를 반영하지 않음.  
→ 검색으로 이동한 태스크 상세가 목록·상세와 맞지 않을 수 있음.

**필요 조치:**  
태스크 상세가 id를 반영하도록 수정한 뒤, 검색 결과의 task id와 동일한 체계를 쓰면 됨.

---

## 요약 표

| 구분 | 설명 | 우선순위 |
|------|------|----------|
| 워드 "연결된 트랙" | `/suno/projects/{trackId}` → trackId≠projectId | 높음 |
| SUNO 최근 활동 | 프로젝트로 가는 링크 없음 | 높음 |
| 트랙 목록 vs 상세 | 서로 다른 목업 → 같은 id여도 다른 트랙 | 높음 |
| 태스크 상세 | params.id 미반영, 항상 동일 목업만 표시 | 높음 |
| 앨범 | id는 맞지만 데이터 소스 이중화 | 보통 |
| 검색→태스크 | 상세 id 반영 후 검색 id와 통일 | 보통 |

이 문서는 위 항목들을 **연결 끊김** 기준으로만 정리한 것이며, 실제 수정 시에는 API 연동 계획과 함께 데이터 소스 통일을 권장합니다.
